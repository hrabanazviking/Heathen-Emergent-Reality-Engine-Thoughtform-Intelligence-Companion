# Ghost Echo Model

## Purpose

Define ghost echoes as first-class records of non-selected futures.

## Definition

A ghost echo is a persisted record of a viable but non-manifested path considered during collapse.

## Canonical fields

```yaml
ghost_echo_id: "ge_uuid"
thoughtform_id: "tf_uuid"
source_node: "intention"
target_node: "wild_ablate"
created_by_event_id: "evt_uuid"
collapse_sequence: 1042
probability: 0.44
collapse_score: 0.78
measurement_source: "voice_fft"
branch_depth: 2
ritual_context_ref: "session_uuid"
status: "active|compacted|expired|promoted"
influence_weight: 0.18
summary: "brief description of abandoned future"
```

## Uses

- alternate-life recall
- evolution seed material
- forensic replay
- divergence analysis
- speculative advisory generation

## Promotion paths

A ghost echo may be promoted into:
- reflection context
- alternate future query result
- mutation seed
- resurrected fork candidate

## Retention classes

- hot recent echoes
- compacted summary echoes
- lineage-critical echoes
- disposable low-signal echoes

## Rules

1. Ghost echoes are never treated as manifested history.
2. Promotion must leave original provenance intact.
3. High-value ghost echoes should survive compaction as summaries at minimum.
4. Cross-thoughtform ghost echoes require lineage-safe references.

## Acceptance criteria

- A user can inspect a thoughtform’s major abandoned paths.
- Evolution can consume ghost echoes without overwriting history.
- Storage can be compacted without erasing provenance.
